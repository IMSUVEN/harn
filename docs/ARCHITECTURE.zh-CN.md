# 架构

anima 是一个 Rust CLI 工具，向项目中植入可生长的种子。其架构是文档、种子与交付它们的 CLI 三者之间的关系。

## 文档层次

```
  哲学（anima 特有）
       |
       | 诠释，不覆盖
       |
  规范 + 指南（学科层面）
```

**Harness 规范**与 **Harness 指南**将 harness 工程表述为学科——一般性理论，取控制视角。**哲学**以培育视角重释同一套公理。这是单向依赖：哲学建基于规范/指南，但规范/指南独立于 anima。

## 种子

```
  哲学 §4（种子应当是什么的理论）
       |
       | 具现为
       |
  seed/（anima init 实际产出的文件）
```

`seed/` 目录包含 `anima init` 植入用户项目的确切文件。唯一的参数化值是 `seed/AGENTS.md` 中的 `{project-name}`。种子实现了哲学 §4，并通过培育协议回应 §6.2 所揭示的记忆悖论。

### 种子代际

种子以代际版本化——一个简单的整数，在新增有意义的能力时递增。`seed/AGENTS.md` 末尾的标记 `<!-- anima:seed:N -->` 记录代际。没有标记的种子为第 1 代（该机制存在之前播下的所有种子）。

`anima check` 检测已播种的代际，若落后于当前代际，则输出具体的升级建议。智能体读到这些建议后自行执行——不强制覆写，因为种子文件是项目已经修改过的活文档。这是培育而非控制：观察、建议、让智能体行动。

## CLI

```
  seed/（种子内容的源头）
       |
       | 经 include_str! 嵌入
       |
  src/main.rs（CLI 二进制）
```

CLI 在编译时通过 `include_str!` 嵌入种子文件，因此二进制是自包含的——运行时无需外部文件。

现有两个命令：

- **`anima init`**：写入种子文件并将 `{project-name}` 替换为项目名称（从目录名推断或通过 `--name` 指定）。若 `AGENTS.md` 已存在则拒绝覆写。

- **`anima check`**：读取项目的 harness 文件并报告其培育状态——哪些区域已生长、哪些仍休眠。这是精灵的第一个「感官」：它观察四个信号（状态阶段、架构文档、决定记录、约定），产出一段紧凑的摘要，设计为供智能体在会话开始时使用。

## 工程基础设施

`src/main.rs` 中的单元测试（`#[cfg(test)] mod tests`）覆盖驱动 `anima check` 的解析函数：阶段提取、种子代际检测、约定计数、种子嵌入完整性。这些是精灵的感官器官——必须可靠。

GitHub Actions 提供两条工作流：

- **CI**（`.github/workflows/ci.yml`）：每次推送至 `main` 及每个 Pull Request 时，在三个平台（Linux、macOS、Windows）上运行 `cargo test` 和 `cargo build`。

- **Release**（`.github/workflows/release.yml`）：由版本标签（`v*`）触发。为四个目标（x86_64 Linux、x86_64 macOS、aarch64 macOS、x86_64 Windows）编译发布二进制，打包后发布至 GitHub Releases，附自动生成的发布说明。

## 翻译

`docs/` 中每份文档均有中文翻译（`*.zh-CN.md`）随附。先写英文，再译中文（信达雅标准）。翻译平行维护——英文版的变更须在同一 commit 中对应修改中文版。

## 尚不存在的部分

- **精灵基础设施**：无 `.anima/` 目录、无持久服务、无生态信号处理。这些属于觉醒阶段（§6.4）。
