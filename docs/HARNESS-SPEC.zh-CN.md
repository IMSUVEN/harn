# 智能体软件工程 Harness 规范

[English](HARNESS-SPEC.md) | [中文](HARNESS-SPEC.zh-CN.md)

> 人机智能体系统中固有三种结构性矛盾。它们是恒久存在的——既不是待修复的缺陷，也不是终将克服的局限——任何 harness 都必须同时应对这三者。（姊妹文档 [Harness 指南](HARNESS-GUIDE.zh-CN.md) §1–2 给出完整分析。）
>
> - *意图传递矛盾：* 人的意图含混且依语境而生；智能体执行字面化且绑定语境。这一鸿沟是结构性的。
> - *自我评估矛盾：* 产出工作的同一系统无法客观评估该工作。自利偏差是自我评估本身的属性。
> - *熵矛盾：* 模式复现既是智能体最大的优势，也是其最大弱点。
>
> 这些矛盾是递归的：它们不仅作用于智能体的工作，也作用于 Harness 本身。Harness 将人类意图字面化为规则（意图传递），其设计者是自身设计的偏颇评判者（自我评估），而 Harness 模式会跨项目自行复制（熵增）。因此，下列公理在两个层面生效：既适用于 Harness 内运行的智能体，也适用于其运行语境中的 Harness 自身。（§4.3 规定了 Harness 层面的义务；姊妹文档 [Harness 指南](HARNESS-GUIDE.zh-CN.md) §1 展开了完整的递归分析。）
>
> 下列公理是应对这些矛盾的原则。
>
> **公理一、人掌舵，智能体执行。**
> *应对意图传递与自我评估两类矛盾。* 稀缺资源是人的注意力。应将其投入环境设计，而非逐行写代码。此公理界定权责划分——人定义意图与边界，智能体在边界之内将其实现。
>
> **公理二、从简起步，为复杂度付账。**
> *规定管理机制部署到何种强度。* Harness 的每个组件都隐含对「模型做不到什么」的假设。若模型单凭自身即可胜任，该组件就是浪费。应在每一代模型上检验这些假设。此公理界定增长策略——从回路中的单个智能体起步，仅当当前层级可证明地失败时才增加结构。
>
> **公理三、一切皆回路。**
> *应对熵与自我评估两类矛盾。* 智能体行动 → 环境响应 → 智能体调整。回路设计得当，智能体收敛；设计失当，则漂移。当同类失败反复出现时，应从结构上解决——增加规则、测试或工具——使回路能够自我纠正。此公理界定一切智能体工作的根本机制。

## 0. 前言

### 0.1 本文档的定位

本文档是关于围绕编程智能体（Agent）构建 **Harness（智能体工作环境框架）** 的 **规定性规范**。Harness 指塑造智能体如何工作的环境、反馈回路和控制系统。Harness 不是模型本身，而是模型之外的一切。

**范围。** 本规范适用于软件工程类智能体——即读取、编写、测试和部署代码的智能体。既涵盖从零开始的项目，也涵盖在既有代码库上的渐进式采用。与具体智能体平台无关：无论使用 IDE 内置智能体、自托管编排器还是 SaaS 智能体平台，要求均适用。若某项要求对既有代码库不切实际（例如遗留无类型项目难以做到全面类型覆盖），规范会说明渐进式采纳路径。非软件类智能体（数据分析、内容生成、机器人等）不在范围内。

本规范按 **成熟度级别** 组织：级别 1（单智能体）、级别 2（多智能体）、级别 3（集群与机群）。每一级包含前一级。从级别 1 起步；仅当当前级别无法解决已遇到的瓶颈时再升级。

### 0.2 义务用语

本文档采用 RFC 2119 关键词：

- **MUST（必须）**：强制性要求。违反将导致可预期的失败。
- **SHOULD（应当）**：强烈建议。仅在具有明确理由时方可偏离。
- **MAY（可以）**：可选。在对你有价值时采用。

本规范包含两类义务：

- **结构性义务** 源于人机智能体系统中恒久存在的结构性矛盾。无论模型能力如何均成立，且不预期会改变。例如：「将知识存放在仓库中」（§1.1）、「在系统边界校验输入」（§1.2）、「沙箱化执行环境」（§1.7）。
- **校准性义务** 将当前工程判断编码为关于具体阈值的约定。其底层原则具有持久性，但具体数值反映当下技术水准，应在每一代模型上加以压力检验。当出现具体数字（≤ 1 分钟、≤ 2 秒、约 100 行）时，该义务属于校准性：挑战阈值，而非原则。

后文仅使用英文 **MUST**、**SHOULD**、**MAY** 表示上述含义。

### 0.3 工作流模式速览

在引入智能体之前，选择能覆盖任务的最简模式：

| 模式 | 适用情形 | 级别 |
|------|----------|------|
| **单智能体 + 回路** | 大多数任务；工具体系健全、提示清晰 | 1 |
| **提示链（Prompt chaining）** | 任务可分解为固定的顺序子任务 | 1 |
| **路由（Routing）** | 不同类别需要不同处理（例如简单任务 → 更快模型） | 1 |
| **并行化** | 子任务相互独立；或为置信度对同一任务投票 N 次 | 1–2 |
| **编排者—工作者（Orchestrator-workers）** | 无法预先预测子任务 | 2 |
| **评估—优化（Evaluator-optimizer）** | 有明确评估标准；迭代改进可带来可度量价值 | 2 |
| **规划者 + 生成者 + 评估者** | 复杂、长达数小时的构建，需要规划、执行与质检 | 2–3 |

**默认模式是回路中的单个智能体。** 仅在能证明可改善结果时再升级。

## 1. 级别 1 — 单智能体与回路

级别 1 是基础。单个智能体在单个仓库中运行，每次回路迭代处理一项任务。更高级别均建立在此之上。

**采纳优先级。** 若渐进式采纳，应最先落实、杠杆效应最高的 MUST 包括：仓库知识（§1.1）——使智能体能自我定位；测试（§1.5）——使反馈回路快速且真实；以及安全（§1.7）——防止智能体造成损害。上述就绪后，再按最常出现的失败模式，依次加强代码可读性（§1.2）、机械约束（§1.4）、工具设计（§1.3）与开发环境速度（§1.6）。

### 1.1 仓库知识

智能体只能就其可见内容进行推理。仓库之外的知识，对智能体而言等效于不存在。

**MUST**：在仓库根目录维护 `AGENTS.md`，作为智能体的入口。该文件是 **地图**（约 100 行），而非百科全书。它指向仓库内更深层的权威信息。*原因：单块指令文件会挤占任务上下文且迅速腐化；地图保持稳定，智能体可按需导航到细节。*

**MUST**：将所有权威项目知识——架构、设计决策、约定——以版本化文件形式存放在仓库中。*原因：对话、聊天线程与 Wiki 对智能体不可见。不在仓库里，对智能体而言即不存在。*

**SHOULD**：在 monorepo 中使用嵌套的 `AGENTS.md`，每个子项目一份。离被编辑路径最近的文件优先。*原因：不同子项目的构建命令、约定与约束不同；单一根文件难以承载而不膨胀。*

**SHOULD**：维护 `ARCHITECTURE.md`，描述领域分层与主要模块边界。*原因：智能体需要系统组织方式的心智模型，才能做出与现有结构相符的修改。*

### 1.2 代码可读性

智能体通过文件系统、类型系统与工具接口浏览代码。这些表面 MUST 为智能体理解而设计。

**MUST**：使用具备 **构建期类型检查** 的语言——即在运行前由编译器或构建步骤中的类型检查器校验类型的语言（例如 TypeScript、Go、Rust、在严格模式下配合 mypy/pyright 的 Python、Kotlin、C#）。在系统边界（API 客户端、数据库 schema、模块间契约）上做到端到端类型覆盖。对既有动态类型代码库，采用渐进式类型化：启用严格类型检查器，对新增与修改文件强制完整类型覆盖，再逐步扩大范围。*原因：类型在构建期排除大量非法状态，缩小智能体的搜索空间。无类型边界迫使智能体猜测数据形状，导致静默失败。*

**MUST**：在系统边界将外部输入校验并解析为类型表示（parse, don't validate）；在边界之内信任类型系统。*原因：这样可把校验逻辑集中在少数经充分测试的点上，而非在代码库各处散落防御性检查——智能体难免会遗漏其中一部分。*

**SHOULD**：优先采用多个短小、职责清晰的文件，而非少量巨大文件。*原因：智能体会截断或概括大文件，丢失上下文；短文件可以完整载入。*

**SHOULD**：使用语义化目录路径（`billing/invoices/compute.ts`）与语义化类型名（`UserId`、`SignedWebhookPayload`）。*原因：智能体从名称推断意图。泛化命名（`utils/helpers.ts`、`T`、`data`）会叠加歧义。*

### 1.3 工具设计（ACI）

智能体—计算机界面（Agent-Computer Interface，ACI）值得像人机界面一样投入设计精力。

**MUST**：将工具设计得难以误用（防错，poka-yoke）。例如：要求绝对文件路径，而非相对路径——相对路径在智能体切换工作目录时会失效。*原因：智能体难以从模糊或静默的工具失败中恢复；错误会在后续步骤中放大。*

**SHOULD**：将工具文档当作提示工程来撰写——包含示例用法、边界情况、输入格式要求以及与其他工具的边界。*原因：工具描述是智能体决定如何使用的唯一上下文。*

**SHOULD**：使工具输入/输出格式贴近自然出现的文本（markdown、纯代码）。避免认知负担高的格式（带 chunk 头的 diff、JSON 转义的代码等）。*原因：当格式接近训练分布中的常见形态时，模型生成这些格式更可靠。*

### 1.4 机械约束

**MUST**：通过 linter 与结构性测试落实架构不变式，不能仅靠文档。*原因：智能体会复制既有模式，包括违规模式。未机械落实的文档规则几天内就会被忽视。*

**MUST**：编写自定义 linter 报错信息时包含 **修复指引**。*原因：报错会直接注入智能体上下文；清晰的修复指引能把失败转化为自我纠正。*

**SHOULD**：在层与层之间强制依赖方向（例如 `Types → Config → Repo → Service → Runtime → UI`）。*原因：没有方向约束时，智能体会制造循环依赖，使代码库越来越难推理。*

**SHOULD**：在可能时将主观质量偏好（「品味」）编码为机械规则。当某偏好反复被违反时，将其从文档提升为 linter 规则。*原因：文档是建议；代码是强制。编码后的品味对每一行代码一视同仁。*

### 1.5 测试

**MUST**：使完整测试套件可在 **≤ 1 分钟** 内执行完毕。*原因：智能体在回路中以测试作为反馈。套件过慢意味着智能体在检查之间迭代，错误会堆积。应通过并发、隔离与第三方调用缓存等方式投资测试速度。*

**SHOULD**：对智能体生成的代码库以 100% 代码覆盖率为目标。*原因：在 100% 时，覆盖率报告即明确的待办列表——无需判断「什么值得测」。低于 100% 时，智能体必须判断「重要程度」，引入决策面并系统性地欠测。这是相变而非边际改进。对大型既有代码库的增量修改，可改为对变更行强制覆盖率。*

### 1.6 开发环境

**MUST**：使开发环境可通过 **一条命令、≤ 2 秒** 拉起。*原因：智能体（及编排器）会频繁创建与销毁环境。手工步骤成为瓶颈，阻碍并行。*

**MUST**：隔离环境，使多个实例可同时运行而不冲突（端口、数据库、缓存、后台任务）。*原因：即便在级别 1，开发者也常在另一 worktree 中手动测试的同时运行智能体。级别 2 及以上并发是硬性需求。*

**SHOULD**：使用 git worktree 做环境隔离，并为每个 worktree 配备可观测性（日志、指标、追踪），在任务结束时销毁。*原因：worktree 在共享 git 历史的前提下提供文件系统隔离；短命可观测性避免脏数据污染后续运行。*

### 1.7 安全基础

**MUST**：在沙箱环境中运行智能体，不得访问生产凭证或客户数据。*原因：智能体执行任意代码；未沙箱且能访问生产的智能体是随时可能发生的安全事件。*

**MUST**：对工具权限（网络主机、shell 命令、文件路径）使用白名单。默认拒绝。*原因：智能体会使用一切可用工具。无限制访问意味着可能误调生产 API、删除仓库外文件或外泄数据。*

**SHOULD**：永远不要把密钥提交进仓库。在沙箱 worktree 中通过环境变量注入。*原因：智能体会读取整个仓库；仓库中的密钥会进入上下文并可能泄漏到输出。*

在级别 1 中，假定人类直接监督。智能体在遇到破坏性或不明确操作时，默认行为 SHOULD 为上报并等待，而非擅自继续。（级别 2 将其形式化为明确的升级协议——见 §2.6。）

## 2. 级别 2 — 多智能体与规划

当单个智能体在上下文窗口内始终无法完成任务、输出质量需要智能体难以可靠自评的迭代评估，或任务范围需要预先分解时，升级到级别 2。

### 2.1 智能体角色

级别 2 引入三种专门角色，分别针对单智能体模式的典型失败模式。

**规划者（Planner）**：将简短（1–4 句）提示扩展为含里程碑的完整产品规格。*原因：没有规划者时，智能体容易低估范围——不经思考完整功能集就直接开写，导致设计不连贯。*

- MUST 聚焦产品语境与高层技术设计，而非琐碎实现细节。*原因：若规划者在实现上过度规定且出错，错误会级联到构建阶段。*

**生成者（Generator，Builder）**：按计划逐项实现功能。

- MUST 在每个工作单元结束时自评，但 MUST NOT 作为质量的最终裁判。*原因：自评能发现机械性错误，但在主观质量与完整性上存在自利偏差。*

**评估者（Evaluator，QA）**：与生成者在结构上分离。测试运行中的应用，并按明确标准评分。

- MUST 与生成者处于 **不同的智能体会话**。*原因：大语言模型在评估自身输出时存在自利偏差——倾向正面评价并为缺陷辩解。结构上独立的评估者更容易校准为怀疑态度，这与代码评审优于自我评审是同一道理。*
- MUST 实际操作 **运行中的产物**——不仅阅读源码。对 Web 应用指浏览器自动化与 API 测试；对 CLI 工具指运行命令并核对输出；对库指针对公共 API 执行集成测试。*原因：能编译且看似正确的代码仍可能不可用。只有实机交互才能暴露集成失败、工作流断裂与行为缺陷。*
- SHOULD 通过带详细分数分解的少样本示例进行校准。*原因：没有校准时，评估者会随时间趋向宽松。*

**智能体间评审回路**：生成者完成一个工作单元后，由另一智能体会话（通常为评估者或另一生成者实例）在合并前评审代码。流程为：生成者构建 → 生成者自评 → 独立智能体评审 → 根据反馈迭代 → 合并。人类评审保留给真正需要判断的情形。*原因：智能体评审吞吐远超人类。把人类留给高杠杆判断，使其聚焦最关键决策。*

### 2.2 执行计划（ExecPlan）

ExecPlan 是级别 2 的主要协调工件。它将非正式任务描述替换为结构化、持久、自包含的文档。

**MUST**：使每个 ExecPlan 完全自包含。仅持有计划与工作树的初学者也必须能成功。*原因：上下文重置（§2.4）会销毁一切会话状态。ExecPlan 是唯一幸存物。若引用新会话上下文之外的文档，计划即失效。*

**MUST**：将 ExecPlan 视为活文档。工作推进时更新 Progress、Decision Log 与 Surprises 等章节。*原因：计划会被没有其它上下文的后续智能体会话消费。陈旧计划会导致重复劳动或与已定决策矛盾。*

**MUST**：将验收标准表述为 **可观察行为**（例如「访问 `/health` 返回 HTTP 200」），而非代码结构（例如「增加了 HealthCheck 结构体」）。*原因：智能体会针对字面验收标准优化。若标准是结构性的，智能体可能只凑出结构而不保证可用。*

**MUST**：将步骤设计为幂等——重复执行不致破坏。*原因：智能体会话可能在步骤中途崩溃。下一会话会重新执行该步骤；若非幂等，会产生重复或损坏。*

完整 ExecPlan 模板见 [附录 A](#appendix-a-templates)。

### 2.3 Sprint 契约

在每个工作单元之前，生成者与评估者 SHOULD **协商 Sprint 契约**：就「完成」的含义达成明确一致，包括可测试的验收标准与验证方法。

*原因：产品规格有意保持高层。Sprint 契约弥合用户故事与可测试实现之间的鸿沟，避免生成者做错对象或评估者按未言明的期望打分。*

Sprint 契约模板见 [附录 A](#appendix-a-templates)。

### 2.4 上下文策略

**SHOULD**：当模型开始退化——过早收尾、丢失计划线索或质量下降——优先采用上下文重置（清空窗口、以结构化交接启动新智能体），而非仅靠压缩。*原因：压缩保留连续性，但不给智能体干净起点。表现出「上下文焦虑」的模型在压缩后往往仍会如此。*

**MUST**：执行上下文重置时，离开的智能体 MUST 产出结构化交接工件，内容包括：已完成工作、当前状态（应用状态、通过测试、已知问题）、有序后续步骤，以及继任者所需关键上下文。*原因：没有交接时，继任智能体从零开始，重复已完成工作并与先前决策矛盾。*

交接工件模板见 [附录 A](#appendix-a-templates)。

### 2.5 质量标准

**MUST**：定义明确的评分标准，并向生成者与评估者提供同一套标准。标准 MAY 定义在 Sprint 契约（§2.3）中、ExecPlan 的里程碑定义中，或独立文档中——机制灵活，但标准本身 MUST 存在且 MUST 共享。*原因：生成者需要知道目标；评估者需要知道记分卡。标准不一致会产生「生成通过、评估失败」的工作，浪费迭代。*

**MUST**：为每项标准设定硬性通过/不通过阈值。任一项不达标即 Sprint 失败，生成者获得详细、可操作的反馈。*原因：软阈值让评估者能为边缘通过找理由。硬阈值迫使生成者真正修复问题。*

**迭代改进**：生成者构建 → 评估者测试实机应用 → 打分与批评 → 迭代（1–5 轮）。若方向根本错误，生成者 SHOULD 整体转向，而非在错误方向上抛光。

全栈应用建议标准：

| 标准 | 权重 | 失败时的表现 |
|------|------|----------------|
| **产品深度** | 高 | 功能仅为展示壳，实际不可用 |
| **功能性** | 高 | 真实用户路径下存在缺陷；工作流断裂 |
| **视觉设计** | 中 | 识别度不一致、版式损坏、无响应式 |
| **代码质量** | 中 | 无测试、无错误处理、违反架构 |

### 2.6 级别 2 的安全

**MUST**：定义人类升级协议——明确智能体停止并询问的触发条件：
- 破坏性操作（丢数据的迁移、force-push、生产部署）
- 计划无法解决的模糊或矛盾需求
- 同一步骤重复失败（3 次以上且无进展）
- 安全敏感变更（认证、支付、访问控制）
- 成本或时长上限超出

*原因：没有明确的停止条件时，智能体要么静默卡住，要么在高风险操作上即兴发挥。两者都不如主动询问。*

智能体 MUST 报告：尝试了什么、被什么阻塞、建议下一步。不能只说「卡住了」。

## 3. 级别 3 — 集群与机群

当待办持续超出单条智能体流水线（规划者 → 生成者 → 评估者）的处理能力，且需要在多个智能体间并行执行时，升级到级别 3。

### 3.1 工作流持久性

级别 2 引入了持久性原语：ExecPlan（§2.2）、交接工件（§2.4）与幂等步骤。在级别 3，这些原语被 **工业化**：会话频繁过期，多个智能体可能并发处理同一计划的不同步骤，任意智能体必须能在任意时刻恢复任意工作流。这需要 **非确定性幂等**：工作流路径非确定性（由智能体决定如何做），但结果在验收标准意义上确定性（因工作状态持久）。

**MUST**：将所有工作状态存放在智能体会话之外——git（ExecPlan、Sprint 契约、交接）或 issue 跟踪系统中。*原因：智能体会话本质上是短暂的。若状态仅在会话内，崩溃或上下文耗尽会丢失全部进度。*

**MUST**：为每个工作流步骤给出明确完成条件。智能体认领步骤、执行并标记完成。进度始终在计划的 Progress 章节中可见。*原因：没有明确完成标记时，下一智能体无法判断从何处恢复，导致重复或遗漏。*

**MUST**：使步骤认领具有原子性——同一时刻仅一个智能体处理给定步骤。*原因：没有独占认领时，两个智能体可能并发处理同一步骤，产生冲突变更。*

**会话如牛群般可替换；工作工件须持久。** 智能体会话可互换——任意智能体通过读取持久状态即可接手进行中的工作流。

### 3.2 伸缩架构

- **集群**（5–30 个并发智能体）：多个智能体并行从同一 backlog 取任务。MUST 配备监督/见证智能体监控工作者健康。*原因：无监督时智能体会卡住、静默停滞或死循环；监督者负责检测并重启。*
- **机群**（100+ 智能体）：监督智能体管理编码智能体 pod，仅在真正卡住时升级到人类。

### 3.3 合并队列

**MUST**：运行专用的合并智能体（或「精炼厂」），按队列顺序处理合并，每次一个 PR。*原因：多智能体并行时，群体工作期间基线剧烈变化。朴素并发合并会带来持续冲突与构建失败。*

**MUST**：合并智能体 MUST 在语义上理解冲突双方，而非盲目 rebase。*原因：后到智能体可能需要在显著不同的 HEAD 上重新构思变更。机械 rebase 会丢失意图。*

**SHOULD**：以车队式工单跟踪并行智能体产出，展示进行中、已合并与卡住项。*原因：机群规模下，交付状态的可视化对人类监管者排优与解阻至关重要。*

### 3.4 高级安全

**MUST**：Harness 产出的每次部署 MUST 具备可回滚能力（蓝绿、金丝雀或特性开关）。*原因：机群规模下变更量使人工验证不可行。自动回滚是安全网。*

**SHOULD**：部署后立即运行自动化冒烟测试。失败则自动回滚或升级。*原因：部署与发现问题之间的时延决定爆炸半径。*

**SHOULD**：为每次合并打上便于 revert 的提交。*原因：单命令 revert 是最快的恢复路径。*

## 4. 横切关注点

以下关注点适用于所有成熟度级别。*原则* 普遍适用；*实现* 随级别扩展。

### 4.1 熵管理

智能体生成的代码会复制既有模式——包括次优模式。若不主动维护，代码库会滑向不连贯。

**MUST**：主动管理熵——在偏差复合之前检测并纠正模式偏离。*原因：技术债会复合。持续小修远比周期性大扫除便宜。*

- *级别 1*：在日常人类监督中审查智能体输出是否偏离模式。当某偏差反复出现，将修复提升为 linter 规则（§1.4）。
- *级别 2+*：按固定节奏运行后台智能体，扫描与黄金原则的偏差、按域更新质量等级，并开启针对性重构 PR。

**SHOULD**：在 `tech-debt-tracker.md` 中跟踪已知债务；在 `QUALITY_SCORE.md` 中维护质量等级。*原因：没有度量时，漂移在酿成危机前不可见。*

### 4.2 知识卫生

**MUST**：使文档与代码库同步。过时文档会主动误导智能体，比没有文档更糟。*原因：文档比代码腐化更快。智能体按陈旧说明操作会产生自信的错误输出。*

- *级别 1*：在相关代码变更中一并审查文档。行为变更时，在同一提交中更新对应文档。
- *级别 2+*：运行定期的文档整理智能体，扫描陈旧文档并开启修复 PR。

**SHOULD**：当团队对话产生架构决策时，将其固化为仓库工件。*原因：聊天中的决策对智能体与未来成员不可见。不在仓库里，就会被违背。*

### 4.3 Harness 演进

三种基本矛盾是递归的——它们也作用于 Harness 自身（见前言）。因此 Harness 必须在自己的反馈回路中运转：约束智能体、观察结果、调整自身。这一回路比智能体回路更慢（以周而非分钟计），但同样需要三个属性：可观察的结果、诚实的度量、结构性的纠偏。

**SHOULD**：定期跟踪可观察的 Harness 健康指标——不仅限于模型升级之后。有用指标包括：智能体自主完成率（无需人工干预即完成的任务占比）、缺陷逃逸率（智能体宣告完成后才发现的问题占比）、以及 Harness 组件触发频率（每条规则被触发的频次——从不触发的规则可能已过时；频繁触发的规则可能需要重新设计）。具体指标属校准性义务；度量行为本身属结构性义务。*原因：没有健康指标的 Harness，会遭受它本应防止在代码库中出现的同类隐性漂移。*

**SHOULD**：依据结果数据而非设计直觉来评估 Harness 变更。修改 Harness 时，使用一致的任务集对比前后的智能体输出质量。*原因：自我评估矛盾同样作用于 Harness 设计者。结果数据是 Harness 层面的「结构性独立评估者」等价物。*

每个 Harness 组件都编码了对模型局限的假设。新模型发布时：

1. 每次移除一个组件。
2. 度量对输出质量的影响。
3. 仅保留真正承重的部分。
4. 增加新组件以利用新能力。

两类 Harness 组件生命周期不同：

**脚手架**——补偿当前模型弱点。模型改进后可弃用：

| 组件 | 可弃用时机 |
|------|------------|
| Sprint 分解 | 模型能在完整构建中保持连贯 |
| 上下文重置 | 仅靠压缩即足够 |
| 独立评估者 | 模型自评变得可靠 |

**工程基础**——无论模型能力如何均有价值。永久保留：

| 组件 | 为何永久 |
|------|----------|
| 机械约束（linter、结构性测试） | 熵是系统固有属性，非模型局限 |
| 100% 测试覆盖 | 该约束消除决策负担，与模型测得好坏无关 |
| ACI 防错 | 模糊界面对任何智能体（包括超强模型）都会致错 |
| 类型驱动边界 | 类型同时是文档与编译器强制正确性 |

有趣的 Harness 组合空间不会随模型变强而缩小，而是 **迁移**。

### 4.4 技术选型

**SHOULD**：偏好「无聊」技术——可组合、API 稳定、训练数据中代表性好。*原因：智能体对熟悉技术建模更可靠。文档稀薄或 API 不稳定的异类依赖会提高每任务错误率。*

**MAY**：为不透明依赖重新实现所需子集，紧密集成并充分测试。*原因：可读、经测的库内实现比不透明上游库更便于智能体维护。*

### A.1 ExecPlan 模板

```markdown
# ExecPlan: <Short, Action-Oriented Title>

Living document. Update Progress, Surprises, Decision Log, and Retrospective
as work proceeds. This plan MUST be self-contained — a reader with only this
file and the working tree must be able to succeed.

## Purpose
What someone gains after this change. How they can see it working.

## Context and Orientation
Current state as if the reader knows nothing. Key files by full path.
Define non-obvious terms.

## Milestones
Each: scope, what exists at the end, commands to run, observable acceptance.
Include prototyping milestones to de-risk unknowns.

### Milestone 1: <Name>
...

## Validation and Acceptance
How to exercise the system. Observable behavior with specific inputs/outputs.
Exact test commands and expected results.

## Progress
- [x] (YYYY-MM-DD HH:MMZ) Completed step.
- [ ] Incomplete step.

## Surprises & Discoveries
- Observation: ...
  Evidence: ...

## Decision Log
- Decision: ...
  Rationale: ...
  Date: ...

## Outcomes & Retrospective
Outcomes, gaps, lessons learned.

## Interfaces and Dependencies
Libraries, types, function signatures that must exist post-implementation.
```

## 附录 B：反模式

| 反模式 | 失败模式 | 修复 |
|--------|----------|------|
| 巨型单体 `AGENTS.md` | 挤占任务上下文；数周内腐化 | 地图（约 100 行）+ 结构化 `docs/` |
| 仅以智能体自评作为 QA | 自利偏差；智能体褒扬自身工作 | 结构上分离的评估者（级别 2） |
| 实现计划过度规定 | 规划者错误级联到构建 | 高层规格 + 协商的 Sprint 契约 |
| 工作状态仅在智能体上下文中 | 会话崩溃 = 进度尽失 | git 中的持久化工件 |
| 过早多智能体 | 非确定性在智能体边界复合 | 先单智能体回路；有证明需要时再扩展 |
| 大文件 + 泛化命名 | 智能体截断内容；无法从路径推断意图 | 小文件、语义化路径与类型名 |
| 系统边界无类型 | 智能体猜测数据形状；静默失败 | 在边界 parse；端到端类型 |
| 测试套件过慢（>1 min） | 智能体不够频繁运行检查；错误堆积 | 通过并发与缓存目标 ≤1 min（§1.5） |
| 无人类升级触发条件 | 在破坏性或模糊操作上即兴发挥 | 定义明确的停止并询问条件 |
| 静态 Harness 设计 | 对模型局限的假设过时 | 每一代模型重新评估组件 |
