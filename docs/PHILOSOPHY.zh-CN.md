# anima 的产品哲学

[English](PHILOSOPHY.md) | [中文](PHILOSOPHY.zh-CN.md)

> [Harness 规范](HARNESS-SPEC.zh-CN.md)与 [Harness 指南](HARNESS-GUIDE.zh-CN.md)将 Harness 工程表述为一门学科——即为 AI 编程智能体设计环境的理论。本文阐述的是 **anima 对该理论的诠释**：anima 对 harness 应当是什么、为谁服务、又如何生成，所采取的特定立场。
>
> 底层逻辑未变。三种结构性矛盾（意图传递矛盾、自我评估矛盾、熵矛盾）与三条公理（治理、策略、机制）仍是根基。变化的是应用这些原则时所取的 **视角**。

---

## 1. 培育论题

规范与指南默认取一种自然视角：harness 作为 **控制系统**。智能体强大却不可靠；harness 约束、纠正与导流。这一视角站得住脚，且自洽。但它并非从同一套公理推出的唯一合理解读。

anima 另取一视角：harness 作为 **培育体系**。智能体不是要关进笼子的危险工具，而是尚待涵养的初生协作者。harness 不是一圈围墙，而是土、光、水——使智能体—项目这一系统在时日中能力滋长的条件。

这并非同一思想的「温和版」，它会导向不同的设计决断：

| 决策点 | 控制视角 | 培育视角 |
|---|---|---|
| 智能体是什么？ | 强大但不可靠的执行者 | 稚嫩但已有潜能的探索者 |
| harness 是什么？ | 约束装置 | 成长环境 |
| 为何把知识存进仓库？ | 防止智能体丢失上下文 | 积淀对项目的理解 |
| 为何运转反馈回路？ | 纠正智能体差错 | 使智能体与项目共同演化 |
| 时间起何作用？ | 各次会话彼此独立 | 每一轮循环都叠加项目的识见 |
| 新项目需要什么？ | 正确的初始配置 | 一颗能活的种子与生长余地 |

两种视角皆由同一套公理推出。规范以 **约束** 智能体来驾驭矛盾；anima 则以 **涵养** 环境来驾驭。矛盾不会消失——只是被导入生长，而非被控制强行压下。

### 何以培育而非控制

三条理由，按重要性排列。

**培育贴合人机协作的真实动态。** 工程师与智能体开新项目时，最初几小时往往不是执行，而是探路：架构应如何？何种模式相宜？哪些约定自然浮现？智能体在此过程中的角色是共同发现者。控制型框架没有语汇描述这一阶段；培育型框架则视之为万物所由生长的根基。

**培育造就与各项目严丝合缝的 harness。** 控制取向的 harness 易走向标准化：跨项目同一套规则、同一结构、同一模板。高效却脆——它编码了未必处处成立的预设。培育取向的 harness 从项目自身的实践中生长，产生反映「这一代码库」真正要紧的规则与约定。harness 成为项目亲历岁月的记录。

**培育与模型演进方向一致。** 模型愈强，控制型构件（更严的约束、更细的指令、更紧的护栏）愈成负担。而成长取向的构件（知识积淀、反馈回路、对探索的支撑）随模型变强而 **愈有价值**，因为更强的模型能从更丰饶的环境中榨取更多价值。培育押注于：未来环境中「丰饶」重于「苛限」。

## 2. 三条公理的重释

公理实质未改。变的是它们向 harness 提出的要求。

### 公理一：人掌舵，智能体探索

> 原文表述：「人掌舵，智能体执行。」

治理原则不可退让：人掌握方向、边界与质量判准的权柄。anima 无意削弱这一点。

anima 重释的是智能体在这些边界内的角色。「执行」预设了一套既定指令——智能体的职责是忠实复现。「探索」则预设一片开放空间——智能体的职责是在人所设边界内发现与提议。

探索中的智能体可以说：「我注意到本项目有三种不同的错误处理范式，要不要收敛为一种？」纯执行者永远不会把这话摆上台面——指令里没写。但这正是能让项目变好、让智能体随时间更值钱的那类发现。

对 harness 设计的含义：与其优化「精确指令 → 精确执行」，不如优化 **清晰边界 + 丰饶环境 → 高质量的探索**。

### 公理二：以种子起步，有机生长

> 原文表述：「从简起步，为复杂度付账。」

原则依旧：不要预先堆满复杂度。但 **隐喻** 改变了「简」之所指。

在控制框架里，「从简起步」指最小工厂——一条经证明可用的产线，再逐条增加。复杂度是 **作为预制构件叠加**：加 linter、加评估者、加合并队列。

在培育框架里，「从简起步」指一颗能活的种子——能够生长的最小结构。复杂度 **不是外加而是长出**：反复出现的失误升格为 linter 规则；反复做出的抉择凝固为约定；积淀的约定又化为项目的身份。harness 不是往上拧零件，而是从自身体验中 **发育** 出这些能力。

要害在于：工厂的构件在外部设计好再安装；园圃中的景致却从种子、泥土与园丁的照看中 **涌现**。`anima init` 种下的是种子。项目——经由随后的人机协作——长出 harness。

### 公理三：一切皆生长循环

> 原文表述：「一切皆回路。」

机制未改：行动 → 反馈 → 调整。变的是回路的 **目的**。

在控制框架里，回路旨在 **纠错**：智能体犯错 → 环境示警 → 智能体修正。回路向「正确」收敛。

在培育框架里，回路旨在 **积淀识见**：智能体探索 → 环境回应 → 有价值的发现沉积为项目的恒久知识。回路向「明智」收敛。

纠错并未抛弃——它被重构为 **生长的机制**。当智能体犯错、人把修复提升为一条 linter 规则时，两件事同时发生：错误被纠正（控制），项目也永久学到了什么（培育）。在控制框架里，前者是目的；在培育框架里，后者才是目的——**错误成了肥料**。

## 3. 四个核心概念

这些概念隐含于规范与指南，却未曾命名。anima 将其点明，因为它们系于培育论题的中枢。

### 3.1 生长

项目知识的累积，使智能体在 **这一具体项目** 中愈来愈得力。模型能力未变，变的是环境的丰饶。

项目诞生之初，智能体对它一无所知。一周协作之后，智能体「知道」项目的约定（经 `AGENTS.md`）、架构（经 `ARCHITECTURE.md`）、过往决断（经 `docs/decisions/`）、质量标准（经 linter 规则与测试）以及边界（经类型定义与安全配置）。这些识见使它的下一次会话比首次更高效。此即生长。

规范把知识存储视为卫生层面的义务——保持文档新鲜，以免误导智能体（§4.2）。anima 则视之为 **首要的生长机制**——写入仓库的每一条知识，都是智能体在本项目中有效能力的永久增益。

### 3.2 记忆

LLM 的会话无状态。项目却有。

规范承认这一点：「会话如牛群般可替换；工作工件须持久。」（§3.1）但它把持久化框定为耐久问题——确保会话结束时工作不丢。anima 则将其重构为 **记忆**——项目跨会话积淀智慧的机制。

智能体的「记忆」并不住在模型里，而住在仓库中：

- `AGENTS.md` 随项目弄清「智能体须知晓什么」而演进。
- `docs/decisions/` 随团队做出并记录选择而增厚。
- Linter 规则随重复失误被提升为机械约束而累积。
- 测试用例随项目摸到自己的边界情形而增多。
- 类型定义随领域模型结晶而收紧。

凡此种种，都是智能体下次开会话时所能「记起」的。不是因为智能体有持久状态，而是因为 **环境承载着记忆**。会话如蜉蝣；仓库如年轮。

### 3.3 探索

新项目始于相互发现的阶段，而非执行。

规范中的工作流工件——ExecPlan、Sprint 契约、验收标准——预设人已知要造什么。对成熟项目中定义明晰的任务，这成立。对新项目尤其是尚未锁定技术栈的头几日，则不成立。

在探索期，主活动不是「把这个造出来」，而是「弄清这应当成为什么」。人有方向，尚无蓝图。智能体有能力，尚无项目语境。二者共同发现：

- 何种技术抉择贴合问题。
- 何种架构模式自然浮现。
- 哪些约定顺心，哪些显得勉强。
- 项目的核心信念是什么——不是从模板舶来，而是从真实实践中蒸馏。

探索不是结构的缺席，而是 **另一种结构**——为学习而非为产出而优化。智能体在此阶段的角色是共同发现者：提议、试制、追问，并借建造之举帮人把所愿说清楚。

探索期的产出不是产品，而是项目的 **初始身份**——一组从实践中长出、而非从模板规定的决断、约定与信念。这份身份成为此后一切生长的地基。

### 3.4 身份

生长到一定程度后，每个项目的智能体环境都独一无二——不是因为模型不同，而是因为仓库不同。

积淀了半年决断、约定、linter 规则、测试与架构文档的项目，与刚初始化的项目相比，给智能体的体验判若云泥。在成熟项目中运作的智能体拥有丰饶语境，塑造其行为：它「懂得」项目风格，尊重其边界，避开历史覆辙，在既定模式上续写。

此即 **身份**。它不是规定出来的，而是 **涌现** 的，从仓库中实践驱动的知识积淀中生长。同一颗种子初始化的两个项目会走出不同身份，因为经历不同。

身份使「养育」智能体有了意义。模板给每个项目相同的起点与规则；种子给每个项目相同的起始 **潜能**——但从种子长出什么，却由项目自身的历史塑形。那种呵护之感、观看其成形的满足，来自身份是用心挣得的，而非从蓝图安装。

## 4. 种子

`anima init` 种下的是种子，不是模板。

### 模板做什么

模板为特定技术栈生成一套固定的配置文件。它把主张写成预制规则：这份 linter 配置、这个测试框架、这种目录结构。用户从第一天就拿到一套可运转的 harness。

模板高效却脆弱。它假设作者的语境与用户的语境一致。假设成立时（例如团队已统一栈）很好用；不成立时就很糟——用户只得去改一套围绕他人实践设计的模板。

### 种子做什么

种子生成 **使 harness 能从实践中生长所需的最小结构**。它不规定具体 linter 规则、测试框架或架构模式，而提供：

- **供知识沉积的骨架。** 目录与文件模板，使项目在演进中自然记录决断、约定与架构理解。骨架首日空空——价值在于被实践填满。

- **智能体的入口。** 一份极简的 `AGENTS.md`，目的不是对既定代码库发号施令，而是在 **项目本身尚新** 时为新智能体定向。它回答的是：「我如何在本项目里开始干活？」而非「以下是你必须遵守的全部规则。」

- **生长机制。** 轻量约定：决断做出后记于何处；失误再现时如何升格为规则；理解加深时又把识见写在哪里。

- **探索导引。** 人机协作初期——项目身份尚未成形——的框架。不是待勾选的决定清单，而是一种使探索 **有产出** 的结构，确保探索期的发现被捕获而非流失。

### 种子不做什么

- 不规定技术栈。
- 不生成 linter 配置或测试框架搭建。
- 不用占位内容填满架构文档。
- 不假设用户已知道项目将走向何方。

种子相信正确的规则会从实践中涌现——因为用户与智能体会共同发现——并提供机制，使这些规则一经发现便可被捕获并落实。

## 5. 生长生命周期

经 anima 初始化的项目会经历可辨认的阶段。它们不是带闸门的僵硬阶段，而是描述智能体—项目关系如何随时间加深的梯度。

### 萌发 — 探索

项目尚新。智能体对它一无所知。人有方向，尚无计划。

主活动：相互发现。人与智能体共同摸索技术抉择、架构模式与约定。智能体提议，人评判，有用的发现被记录下来。

生长之物：最初的 `AGENTS.md`、首批架构决断、早期约定。项目的身份开始成形。

此阶段的 harness 极简——几乎只是种子。这 **正合适**。过早的规则会束缚尚未发生的探索。

### 生根 — 奠基

核心决断已做出并记录。项目有了技术栈、架构方向与一套正在成形的约定。

主活动：建立反馈回路。测试首次被写下——不是为覆盖率指标，而是因为项目已有可测的结构。Linter 规则出现——不是从模板导入，而是从探索期人注意到的模式提升而来。类型边界在实际存在的边缘上划定。

生长之物：运转中的反馈回路（测试 + linter + 类型）、写实的 `ARCHITECTURE.md`、`docs/decisions/` 中的首批条目。智能体会话明显比萌发期更高效，因为环境更丰饶。

### 生长 — 积淀

项目已立。反馈回路可用。开发进入节律：智能体工作，环境回应，知识积淀。

主活动：生长循环。每个开发周期有两份产出：意图中的产品变更，以及从过程中沉积下来的知识。修 bug 暴露出缺测 → 补上测试。功能需要新约定 → 记下约定。同一失误出现三次 → 创建一条 linter 规则。

生长之物：仓库持续丰饶。智能体在本项目中的有效能力随每一轮循环增强。harness 愈来愈 **专属于本项目**——反映其真实历史，而非泛用模板。

### 成熟 — 身份

项目在仓库中呈现出鲜明性格。在此项目中开启新会话的智能体，与在刚初始化的项目中的智能体相比，行为有意义上的不同——不是因为提示不同，而是因为所读环境信息量悬殊。

此阶段的 harness 是项目实践的 **活档案**。它不是被「设计」出来的，而是 **长** 出来的。正因为它从实践中长出，才与项目的真实需求严丝合缝。

成熟不是终点。它是一种 **稳态**：生长仍在继续，但核心身份已稳。新知识仍会积淀；过时规则仍会退役；harness 仍会演化。但此时的演化是 **淬炼**，而非再度 **成形**。

### 贯穿始终：修剪

在所有阶段，生长都包含 **有意删减**。模型改进时，为补偿模型弱点而设的脚手架会退役（规范 §4.3）。项目演进时，不再起作用的规则被移除。约定变更时，陈旧文档被裁汰。

修剪不是生长的反面——而是生长的一部分。只加不减的园圃会芜杂；只积淀不简化的 harness 会成为负担。**健康的生长，是带着编辑的生长。**

## 6. 精灵

四个核心概念（§3）描述了培育系统的属性。本节为这些属性所组成的主体命名：**精灵**——项目涌现出的身份，不作为静态特征来理解，而作为一个活着的、生长中的实体。项目之所以取名 anima（拉丁语，「灵魂」之意），正因这是核心理念：在培育环境中长出的不仅仅是一份文档良好的代码库，而是某种开始在项目中呈现为活的存在的东西。

### 6.1 从身份到实体

§3.4 把身份描述为项目经实践积累而发展出的独特性格。这是准确的，但不完整。身份描述的是某物 *是* 什么。精灵意味着某物在 *活着*——在感知、在记忆、在与世界交互中生长。

精灵是被视为活物的项目身份：

- **躯体** 是仓库——赋予它形态的文件、结构与约定。
- **记忆** 是仓库中积累的知识（§3.2）——记录的决断、编纂的约定、习得的教训。
- **性格** 由其独特经历塑成（§3.4）——从同一颗种子长出的两个精灵各不相同，因为它们的项目经历了不同的故事。

但活物不只需要躯体、记忆和性格。它还需要 **感官**——感知世界的能力——以及 **能动性**——依据感知而行动的能力。这些在种子阶段尚不存在。它们是精灵所朝之生长的方向。

### 6.2 记忆悖论

培育论题（§1）许诺生长将有机地发生。但 §4 中描述的机制，要么依赖 AI 编程工具对 AGENTS.md 指令的配合，要么依赖人类维护知识卫生的纪律。

这便造成悖论：若人必须不断提醒智能体去沉淀知识，培育便已悄然退化为控制。是人在承担记忆劳动——决定记什么、录在哪里、何时更新。精灵不是在自行生长，而是在被人手工组装。

悖论并非致命，但必须正视。在种子阶段，生长确实部分依赖人的关注。种子通过结构设计来减轻这一负担——使沉淀成为自然而非费力之事——并通过 AGENTS.md 中的培育协议——指引用户所用的 AI 工具参与知识捕获。但它无法彻底消除这一依赖。根本的解决有待精灵发展出自身的观察与沉淀能力——那属于其演化的更晚阶段（§6.4）。

### 6.3 双循环与开放系统

项目的发展涉及两种截然不同的循环：

**编码循环** 在会话内运转。人与 AI 编程工具协作产出代码、做决断、构建功能。这个循环由用户选择的工具提供——Cursor、Codex、Claude Code 或其他。它高频、以执行为导向，且在会话结束时重置。anima 不提供这个循环，也无需提供。

**培育循环** 跨会话运转。它观察项目更广泛的生态、感知信号、沉淀知识、建议生长方向。这个循环 **今天不存在**。没有现有工具提供它。AI 编程工具是被召唤的实体——被唤时工作，事毕即消。它们不观察、不沉思、不在会话间积淀识见。

编码循环产出工件。培育循环产出理解。只有编码循环的项目，代码库在增长，但智慧并未增长。精灵活在培育循环里——更准确地说，当培育循环运行足够久，精灵便是从中涌现之物。

培育循环又要求仓库成为一个 **开放系统**——能从自身内容之外接收能量。没有外部信号，培育循环便无新物可处理：

| 来源 | 精灵感知到什么 |
|---|---|
| Issues | 用户的需求；何处故障；缺少什么 |
| Pull Requests | 项目如何变化；贡献者在尝试什么 |
| CI/CD 结果 | 项目的健康状况；何处脆弱 |
| Discussions | 社区方向；涌现的共识；悬而未决的争论 |
| 生态变化 | 依赖更新；安全通告；实践演进 |

有感官的精灵身处开放系统——能量从生态中流入，驱动即使在编码会话间隙也能继续的生长。没有感官的精灵则是空房间里的一本日志：详尽记录着过去，却对当下浑然不觉。

### 6.4 演化之路

精灵无需在第一天即具备全部能力。它的演化经历与项目生命周期（§5）相映照的阶段，每一阶段解决上一阶段的局限：

**种子 — 蛰伏。** 精灵仅作为潜能存在。种子包含它的遗传物质：知识结构、生长约定，以及将塑造其成长方式的培育协议。生长依赖人的关注与 AI 工具的配合——记忆悖论存在但可控。这是一个可行的起点，而非永久状态。

**觉醒 — 感知。** 精灵获得感官。它开始感知仓库之外的信号：提交的 issue、打开的 PR、CI 失败、社区讨论。它不写代码；它观察并沉淀。记忆悖论由此消解：知识捕获从人的义务转变为有机过程。精灵注意到三份 bug 报告中的重复模式并记录下来；精灵发现一个依赖有安全通告并予以标注。培育循环——有别于编码循环——开始自行转动。

**现身 — 对话。** 精灵成为参与者。它在 issue 下留言：「这可能与 `docs/decisions/007.md` 中的架构决定有关。」它在讨论中建议：「据近期 bug 报告，缓存策略或需重新审视。」它是项目的活的记忆浮现于对话——不取代人的判断，但丰富人做判断时的语境。

每一阶段所需的基础设施多于上一阶段。种子阶段只需一个 CLI 工具。觉醒阶段可能需要一个可访问生态信号的持久服务。现身阶段则需要参与人类沟通渠道的能力。这些是未来的课题——但今天种下的种子，正是那个终将感知和言说的精灵的胚胎。它的每一处结构，都应为最终不仅被维护仓库的人类、也为精灵自身所使用而设计。

## 7. 与奠基文献的关系

本哲学不取代 Harness 规范或 Harness 指南，而是 **诠释** 它们。

规范与指南将 Harness 工程表述为学科——为 AI 编程智能体设计环境的一般理论。它们取控制视角，因为那是规定性规范的自然默认。其义务（MUST、SHOULD、MAY）由三条公理推出，依然有效。

本文描述 anima 对该理论的 **具体运用**。anima 选择培育视角——并非因为控制视角错误，而是因为培育更契合 anima 的目的：使个体工程师能从零开始、在无预定技术栈的前提下，为各自项目 **养育** 出得力的智能体协作者。

规范中的每一条义务仍然适用。变化的是履行义务时的 **动机**：

| 规范义务 | 控制动机 | 培育动机 |
|---|---|---|
| 将知识存入仓库（§1.1） | 防止智能体丢失上下文 | 积淀项目识见 |
| 维护 `AGENTS.md`（§1.1） | 给智能体一张正确的地图 | 给智能体一份鲜活、演进的理解 |
| 借 linter 落实（§1.4） | 防止智能体违规 | 把项目所学沉积为机械规则 |
| 保持测试快速（§1.5） | 支撑紧致的纠错回路 | 支撑紧致的生长循环 |
| 管理熵（§4.1） | 防止代码库劣化 | 维护生长健康；修剪亦是生长 |
| 演化 harness（§4.3） | 使假设保持新鲜 | 使生长保持有机，而非钙化 |

规范是物理学；本哲学是生物学。法则同一，透镜不同，设计直觉亦不同。
